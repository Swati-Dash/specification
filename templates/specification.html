Abstract

This specification defines the format for how a planning authority should provide information about {{ item["plural"] }} as data.

This technical specification is accompanied by guidance, examples and other tools providing feedback to data providers.

Status of this specification

This is a draft specification, following the standards for planning data process [process]. The contents of this specification are currently under development, and liable to change based on feedback.

This specification is intended to be published as one of a number of official data standards for the provision of planning data under the proposed Levelling-up and Regeneration bill [LURB], as introduced to parliament on 19 May 2022.

! Warning It is inappropriate to cite this document as other than a work in progress.

Comments and feedback on this specification may be provided on the GitHub issue [TBD] or sent to digitalland@communities.gov.uk.

Conformance

As well as sections marked as non-normative, all authoring guidelines, diagrams, examples, and notes in this specification are non-normative. Everything else in this specification is normative.

The key words MAY, MUST, NOT, and SHOULD in this document are to be interpreted as described in [RFC2119] when, and only when, they appear in all capitals [RFC8174] as shown here.

Documentation page

A data provider MUST provide a list of all of the {{item["plural"]}} within its jurisdiction on a documentation page.

The documentation page MUST be accessible as HTML from a public URL.

The documentation page MUST be clearly owned and operated by the data provider. Where the data provider is a local authority or other public body, the web page MUST be on a .gov.uk domain.

The documentation page MUST include:

  • A hyperlink to the conservation area dataset
  • A clear licence for reusing the data Open Government Licence version 3
  • Any attribution needed when using the data, for example Crown Copyright.

Conservation ares

The list of conservation areas MUST be published in an open data format.

There MUST be at least one entry for each conservation area. Each entry MUST containing the following information:

  • Reference – the short reference for the conservation area
  • Name – the official name for the conservation area
  • Geometry – the boundary for the conservation area
  • Start date – the date the conservation area was adopted.
  • End date – the date the conservation area was withdrawn or blank if the conservation area is active.
  • Entry date – the date this entry was created or updated
  • Documentation url –

The URI is inferred by a combination of the reference and the or be provided in an optional URI field.

List of conservation area documents

  • Reference – a short reference for the document
  • Name – the title of the document
  • Start date – the date the document was published.
  • End date – the date the document was withdrawn, or superseded by another document. This SHOULD be blank if the document is still relevant to planning.
  • Entry date – the date this entry was created or updated
  • Document type – the type of the conservation area document which MUST be one of the following values:
    • “area-appraisal”
    • “notice”
  • Documentation URL – a link to a web page which contains or links to the document
  • Document URL – a link to the document

Datatypes

Date

Dates MUST be in ISO 8601 as format as by the open standards for government guidance on formatting dates and times in data.

A date value SHOULD be blank if it is unknown. The date may just contain the year “YYYY” if only the year is known, or “YYYY-MM” if only the year and month is known.

Geometry

The boundary for the conservation area as a single POLYGON or MULTIPOLYGON value. Points MUST be in the [WGS84] coordinate reference system.

The shape MUST be a valid WKT POLYGON or MULTIPOLYGON when provided as CSV.

Reference

The reference identifies the entity within a data provider.

A reference value MUST be suitable as the reference part of a [CURIE] identifier.

A reference MUST be unique within the dataset; it MUST NOT have been used by the data provider to identify a different entity.

A reference SHOULD be persistent. The same reference should be used by the provider to identify in the future.

The reference MAY be used in other contexts to refer to the entity. For example, the reference may be used to construct a URL to a web page with more information about the {{ item["name"]}}.

A reference SHOULD be short and meaningful to a user. For example, the reference may be used to refer to a {{item["name"]}} when completing a form, or contacting a call centre.

Historical data

Entries SHOULD be ordered by the entry-date, with older entries appearing before later entries.

Open data formats

The data MUST be provided in an open data standard format. The preferred format is Comma Separated Value (CSV).

Alternatively, where the dataset includes a geometry or a point field, the data MAY be provided in one of the following open standard geospatial formats:

  • IETF CSV containing Well Known Text [WKT]
  • IETF GeoJSON
  • OGC Geopackage containing points
  • OGC GML
  • OGC Keyhole Markup Language [KML]

The data may also be provided in other formats such as a zip archive containing a ESRI shapefile collection of files, or Mapinfo TAB file.

In all cases all points MUST be either in the [WGS84] (EPSG::4326) or [ETRS89] (EPSG::4258) coordinate reference system to comply with the Open Standard for government mandate on the exchange of a location point [GOV.UK LOC].

Boundaries provided by this dataset are intended to be used as an index. Positions calculated by the WGS84 and ETRS89 systems currently deviate by less than half a metre for points within England. Survey data SHOULD be used where more precision is needed.
